# Pytra Transpiler Specification: `@extern` / `@abi` and Fixed ABI Types (C++ Backend)

Notes:

- `@extern` is a canonical, currently-implemented feature.
- `@abi` is an approved next-stage extension specification; this document defines the target design.
- Before `@abi` is implemented, generated runtime helpers follow the ordinary internal representation conventions.
- This document is the canonical source for `@abi` syntax / semantics / modes / responsibility separation from `@extern`; the parser/EAST metadata format is fixed separately in `P1-RUNTIME-ABI-DECORATOR-01-S1-02`.

## 1. Purpose

Pytra is a transpiler that converts Python to C++ and other languages. In the C++ backend, object lifetimes are managed using ownership representations such as `rc<T>`.

At the same time, due to transpilation-time optimizations, the same Python type may have multiple C++ internal representations. For example:

- `rc<list<rc<bytearray>>>` -> `list<rc<bytearray>>`
- `list<rc<bytearray>>` -> `list<bytearray>`

This variability is problematic at two kinds of boundaries:

- `@extern` functions, which delegate to external implementations (functions defined on the C++ side)
- Among runtime helpers generated from a pure Python source of truth, functions where we want a fixed ABI form rather than the internal representation

`@extern` functions must always be callable with the same boundary type, regardless of the internal representation at the call site.
Similarly, in generated helpers, there are cases like `str.join` where we want to receive via a fixed value ABI.

This specification resolves this problem by:

- Separating `@extern` (external implementation marker) from `@abi` (generated/helper boundary ABI override),
- Assigning a **fixed ABI type** to each, and
- Inserting **ABI adaptation** at call sites as necessary.

Important:

- `rc<T>` is an internal ownership representation in the C++ backend, not the ABI itself.
- For **value type / container type ABIs** such as `str`, `bytes`, `bytearray`, `list[T]`, `dict[K,V]`, `set[T]`, `tuple[...]`, **`rc<>` must not be exposed**.
- `rc<>` may only be used in the ABI for types where **reference semantics are essential**, such as user classes.
- Internal backend optimizations may use typed handles such as `rc<list<T>>`, but these are **internal types**, not ABI types.
- In particular, when `rc<list<T>>` is used to maintain alias semantics with `cpp_list_model=pyobj`, it must still be normalized to `list<T>` for passing at `@extern` boundaries.

In this specification, `@extern` supports **only no-argument form**.
`@abi` does not replace `@extern`; it is an orthogonal supplementary annotation.

## 2. Terminology

- **Internal Type**
  The actual C++ representation type used during optimization and conversion. Multiple patterns are possible due to demotion.
  Examples: `list<bytearray>`, `rc<list<bytearray>>`, `object`

- **ABI Type**
  The fixed C++ type used at the boundary of `@extern` function calls. Matches the function signature defined in the external implementation (runtime / target language side).

- **ABI Adaptation**
  The process of converting from the internal type at the call site to the ABI type. Inserted only when necessary. Includes the identity transformation (no-op).

- **Value ABI**
  A boundary type that does not include `rc<>`. Examples: `str`, `bytes`, `bytearray`, `list[T]`, `dict[K,V]`, `set[T]`, `tuple[...]`.

- **Reference ABI**
  A boundary type that preserves identity / dynamic dispatch, such as user classes or `object`.

## 3. Basic Specification of `@extern`

### 3.1 Syntax

Only the following are permitted in Python source:

```python
@extern
def write_png(image: list[bytearray]) -> None: ...
```

```python
from pytra.std import extern

@extern
def write_png(image: list[bytearray]) -> None: ...
```

`extern` is simply a marker and behaves as a no-op at Python execution time.

```python
def extern(fn):
    return fn
```

- In the current version (v1), decorated forms such as `@extern(...)` are not supported.
- In v2, this will be split into `extern_fn` / `extern_var` / `extern_class`, with `module` / `symbol` / `tag` as required arguments. See [spec-builtin-functions.md §10](./spec-builtin-functions.md) for details.
- The extern marker for variables is expressed in the form `name = extern(...)`.
- `name: Any = extern()` is treated as an "ambient global variable declaration of the same name".
- `name: Any = extern("symbol")` is treated as an "ambient global variable declaration under an alternate name".
- `name: T = extern(expr)` is treated as a host fallback / runtime hook initialization, as before.
- Ambient global variable declarations are permitted only in v1 for the JS/TS backend; in other backends they are a compile error.
- `@extern` on the runtime source of truth is declaration-only metadata and does not indicate the owner of the native implementation.
- The location of the native owner implementation is determined by the runtime layout / manifest / runtime symbol index.

### 3.2 Semantics

A function decorated with `@extern` delegates to a hand-written implementation provided on the target language side.

Pytra determines the following for `@extern` functions:

- External name (symbol name)
- ABI type

At the call site, even if actual arguments are represented in the internal type, they are converted to the ABI type before calling the external function.

For Python execution compatibility, an `@extern` function may have a Python-executable body (e.g., `return __m.sin(x)`).
The transpiler does not adopt the body of an `@extern` function as the target implementation; it lowers it to an external symbol call.

### 3.2.1 `@extern` Realization Method by Language

The realization method for `@extern` functions differs between C++ and other languages.

**C++**: Emit declaration only + linker binding

- The emitter outputs only the **declaration (prototype)** of an `@extern` function to `.h`.
- Hand-written implementations reside in `src/runtime/cpp/{built_in,std}/*.h` and are combined via `#include`.
- The C++/linker/include system integrates declarations and implementations, so no delegation code is needed.
- Hand-written implementations may freely use templates and overloads (C++-specific features not expressible in EAST3).

**Other languages (JS, Dart, Julia, Zig, PowerShell, etc.)**: Generate delegation code

- For `@extern` functions, the emitter generates **delegation code** to a `_native` module.
- Delegation target: `<module>_native.<ext>` (e.g., `time_native.js`, `math_native.dart`)
- Example of generated code (JS):
  ```javascript
  // std/time.js (generated)
  const time_native = require("./time_native.js");
  function perf_counter() { return time_native.perf_counter(); }
  ```
- `_native` files are hand-written and provide minimal connection code to the host API.
- The delegation cost is assumed to be effectively zero due to compiler optimizations (inlining).

Reason for this difference:
- Hand-written runtimes for C++ make heavy use of C++-specific features such as templates and overloads, making it difficult to auto-generate delegation code from EAST3.
- For other languages, function call delegation is sufficient and can be handled by a unified mechanism.

Variable `extern(...)` is handled separately from function `@extern`.

- `extern(expr)`:
  - Retains `expr` as a Python execution-time fallback / host-only initialization expression.
- `extern()`:
  - Has no Python execution-time fallback; declares an ambient global with the same name as the variable.
- `extern("symbol")`:
  - Has no Python execution-time fallback; declares the ambient global symbol specified by the string.

Ambient global variable declarations are lowered as import-free symbols in the JS/TS backend, and allow raw lowering of property access / method calls / call expressions only for bindings with an ambient global marker, not for general `Any/object` receivers.

`@extern` on runtime source of truth modules is treated as declaration-only metadata.

- `extern_contract_v1` / `extern_v1` only represent the declaration shape of the symbol; they do not encode the location of the native owner implementation.
- The location of the native owner implementation is determined by runtime layout / manifest / runtime symbol index.
- Ambient global variable declarations `extern()` / `extern("symbol")` are a separate lineage from runtime source of truth `@extern`, and must not be mixed into owner determination.

### 3.3 Host-Only Import (`as __name`) Convention

Imports where the alias starts with `__`, such as `import ... as __m`, are treated as host-only imports.

- Host-only imports are used only as Python-execution-time aids (for evaluating `@extern` function bodies in Python).
- This import is not output in the target language code.
- References to host-only imports are permitted only inside `@extern` function bodies or `extern(expr)` initialization expressions.
- If `__m` references appear elsewhere, it is a compile error.
- `_name` (single leading `_`) is not treated as host-only; it is handled as a regular import.

### 3.4 Basic Specification of `@abi` (Approved Extension)

#### 3.4.1 Purpose

`@abi` is used when you want to fix the boundary signature of a generated/runtime helper.

This is distinct from `@extern` in purpose.

- `@extern`
  - Fixes the location of the implementation as "external implementation"
- `@abi`
  - The implementation may be generated or external, but only the ABI form at the boundary is fixed

Therefore, `@extern` must not be abolished by incorporating `no_body` or `external` into `@abi`.

#### 3.4.2 Syntax

```python
from pytra.std import abi

@abi(args={"parts": "value"}, ret="value")
def py_join(sep: str, parts: list[str]) -> str:
    ...
```

In the initial scope, only the following forms are supported:

- Keyword arguments only
- `args={param_name: mode}`
- `ret=mode`

Not supported:

- Positional argument form (e.g., `@abi("value")`)
- Application to classes / methods / lambdas / nested functions
- General public availability to user programs

The initial target may be limited to top-level helpers in runtime source of truth modules (`src/pytra/built_in/*.py`, and `std/utils` as needed).

At Python execution time, `abi` behaves as a no-op decorator.

```python
def abi(*, args=None, ret="default"):
    def deco(fn):
        return fn
    return deco
```

#### 3.4.2.1 Initial Acceptance Conditions

What the parser / validator may accept in the initial introduction is limited to cases satisfying all of the following:

- Application to top-level functions
- Keyword-only form: `@abi(args=..., ret=...)`
- The keys of `args` match existing parameter names
- The canonical mode is one of `default`, `value`, `value_mut`
- The only canonical modes allowed for `ret` are `default` or `value`
- As a migration-period alias, `value_readonly` may be accepted in source surface, but must be normalized to `value` in canonical metadata

All of the following are compile errors:

- Specifying an unknown mode
- Applying an ABI override to an undeclared parameter name
- Positional argument form such as `@abi("value")`
- Application to methods / classes / lambdas / nested functions
- Premature use in general user code other than runtime helpers

#### 3.4.3 Semantics of `@abi`

`@abi` acts not **on the function as a whole**, but **per argument / per return value**.

Rationale:

- No override is needed for immutable arguments like `str`
- We want to fix only mutable containers like `list[str]` to a value ABI
- There are cases where we want a different policy for just the return value

Therefore, the canonical form in the specification is the following per-parameter form:

```python
@abi(args={"parts": "value"}, ret="value")
```

#### 3.4.4 Initial Modes

The modes defined in the initial introduction are the following three:

- `default`
  - No override
  - Follow the existing internal representation / backend default policy
- `value`
  - Fix the return value or argument as the ABI canonical form value ABI
  - In the argument position, means a read-only value ABI
  - The callee must not destructively modify this argument
- `value_mut`
  - Fix an argument as writable value ABI
  - Explicitly marks the rare-case mutable value helper boundary

Notes:

- `value` is particularly important for C++ `list/dict/set/bytearray`.
- This means "does not receive `rc<>`", not "must always copy".
- The C++ backend may use declaration forms like `const list<T>&` / `const dict<K,V>&` where needed, read-only borrowing from the internal handle.
- `value_mut` is a reserved public mode for writable cases; there are currently no checked-in helper usage examples.

The following are out of scope for initial introduction:

- `internal_ref`
- Receiver-dedicated specification

If these become necessary, they will be extended in a separate task.

#### 3.4.4.1 Migration Rules

- The canonical public names are `default`, `value`, `value_mut`.
- `value` on the argument side inherits the meaning of the old `value_readonly`.
- `value` on the return value side continues to mean value return ABI as before.
- `value_readonly` is removed from the canonical surface.
  - However, during the migration period, the source parser may accept it.
  - Even in that case, `FunctionDef.meta.runtime_abi_v1` must normalize it to `value`.
- The old working name `value_mutating` is not adopted. The public name for writable cases is `value_mut`.

#### 3.4.5 Relationship with `@extern`

`@abi` and `@extern` are independent and may be used together.

Examples:

```python
@extern
def sin(x: float) -> float:
    return __m.sin(x)
```

```python
@abi(args={"parts": "value"}, ret="value")
def py_join(sep: str, parts: list[str]) -> str:
    ...
```

```python
@extern
@abi(args={"image": "value"}, ret="value")
def some_native_helper(image: list[bytearray]) -> bytes:
    ...
```

Semantics:

- `@extern` is present
  - The implementation body is not generated; it is lowered to an external symbol call
- `@abi` is present
  - Override the ABI form at the call boundary
- Neither is present
  - Follow normal internal representation conventions and backend default lowering

#### 3.4.6 Necessity for `str.join`

`str.join` in the current C++ runtime is to be handled with a value ABI close to `str::join(const list<str>& parts)`.

However, if the runtime helper is naively "generated-ized", `list[str]` may shift to `rc<list<str>>` due to the C++ backend's ref-first internal model.
This is incompatible with the goal of returning the helper to a pure Python source of truth.

Therefore, helpers like `py_join` are fixed using `@abi` as follows:

```python
@abi(args={"parts": "value"}, ret="value")
def py_join(sep: str, parts: list[str]) -> str:
    ...
```

With this specification, in C++:

- `parts` is handled as a helper that receives the ABI canonical form `list<str>`
- However, the actual function declaration form may be `const list<str>&`
- The callsite inserts a read-only adapter when it sees `rc<list<str>>`

#### 3.4.7 Constraints

- `@abi` is for fixing helper ABI; it does not assign module import / symbol resolution / semantic tags.
- `@abi` must not allow source-side module knowledge to leak to the backend.
- Even if `@abi` is specified, it must not be treated as "external implementation" unless `@extern` is also present.
- If the mode specified by `@abi` and the behavior of the function body are contradictory, it is a compile error.
  - Example: Performing `append` / `pop` / assignment on an argument specified as `value`
- `@abi` is fail-closed; if the backend / lowerer cannot understand the mode, it is a compile error.

#### 3.4.8 EAST / Linked Metadata Format

`@abi` must be retained simultaneously as a decorator surface and as function node metadata.

In raw `EAST` / raw `EAST3`, at minimum the following must be held:

- `FunctionDef.decorators`
  - May retain the raw decorator string as `abi(args={"parts": "value"}, ret="value")`
- `FunctionDef.meta.runtime_abi_v1`
  - Canonical metadata read by backend / linker

Canonical metadata format:

```json
{
  "schema_version": 1,
  "args": {
    "parts": "value"
  },
  "ret": "value"
}
```

Rules:

- `schema_version` is required and must be `1`
- `args` is a map of parameter name -> mode
- `ret` is the return mode
- No canonical mode other than `default`, `value`, `value_mut` is permitted
- `ret` permits only `default` or `value`; `value_mut` is not permitted
- Even when `value_readonly` is accepted at the raw source surface, canonical metadata must normalize it to `value`
- The key order in `args` may be normalized to source parameter order
- When `ret` is unspecified, it defaults to `default`
- When `args` is unspecified, it is an empty map

`FunctionDef.meta.runtime_abi_v1` must be retained after linked-program processing; the linker must not overwrite it.
The linker may only add module-level `meta.linked_program_v1` or call/function summaries; the helper ABI contract itself respects the canonical source from the parser/EAST build.

Parser / selfhost parser acceptance criteria:

- Both backends produce the same `runtime_abi_v1` for the same source
- Unsupported forms are rejected fail-closed during `EAST1/EAST2` build
- Even if the raw `abi(...)` string remains in `decorators`, the backend's canonical source to read is only `meta.runtime_abi_v1`

## 4. External Name (Symbol Name) Resolution Convention

For an `@extern` function `M.f`, the external name is determined as follows.

External name:

```text
pytra_<module>_<function>
```

- `<module>` is the Python module name (e.g., `pytra.std.math`, `pytra.utils.png`)
- `<function>` is the function name

Examples:

- `pytra.std.math.sin` -> `pytra_std_math_sin`
- `pytra.utils.png.write_png` -> `pytra_utils_png_write_png`

Notes:

- The module name determination is unique based on the package name.
- Nested functions and local functions are out of scope.
- `@extern` may be restricted to top-level functions only.
- `@extern` on the runtime source of truth is declaration-only metadata and does not indicate the owner of the native implementation.
- The location of the native owner implementation is determined by runtime layout / manifest / runtime symbol index.
- Ambient global variable declarations `extern()` / `extern("symbol")` are treated as a separate lineage from runtime `@extern`.

## 5. ABI Type Resolution Convention

### 5.1 Basic Policy

The ABI type is a fixed C++ type determined from Python type annotations, and does not fluctuate due to internal optimizations.

The ABI type is treated as the "canonical form at the boundary".

Required rules:

- The ABI **must not expose backend internal ownership representations**.
- `rc<>` is not used in value type / container type ABIs.
- `rc<>` in the ABI is limited to reference types where identity must be preserved, such as user classes.
- `rc<list<T>>` is purely an internal backend representation for alias maintenance and optimization, and is not included in the ABI canonical form.

The `value` / `value_mut` of `@abi` is an annotation for explicitly applying this ABI canonical form to helper boundaries.

### 5.1.1 Backend Internal Representation Default Policy

The ABI canonical form and the representation initially adopted internally in the backend are considered separately.

Required rules:

- For immutable types like `str`, the value type may be used as the default even in the backend internally.
- For mutable types such as `list`, `dict`, `set`, `bytearray`, and mutable user classes, the backend internally defaults to **a representation that preserves shared references**.
- In the C++ backend, `rc<>` or equivalent handles may be used as that shared-reference representation.
- However, that `rc<>` is purely an internal representation and must not be exposed in the ABI.

Value-type demotion is permitted only as an optimization result.

- A mutable value where alias sharing as in `a = b` can be observed must not be demoted to a value type without proof.
- Value-type demotion is limited to cases where safety has been proven by at least mutation / alias / escape analysis.
- Cross-function demotion must be done after building the call graph and fixing summaries per SCC including recursion and mutual recursion.
- When `Any/object` boundaries, `@extern` boundaries, unknown function calls, or unknown-type paths are mixed, fail-closed and retain as shared-reference representation.

In summary: fix the ABI at the value-type canonical form like `list<T>`, while handling mutable types ref-first inside the backend, and only demote to value type afterward on paths where safety has been proven.

### 5.2 Value ABI Canonical Form

Recommended canonical forms in this specification:

```text
bool -> bool

int -> int64
※ In Pytra, Python's int is treated as int64 (signed 64-bit integer).

float -> float64

str -> str

bytes -> bytes

bytearray -> bytearray

list[T] -> list<ABI(T)>

dict[K,V] -> dict<ABI(K), ABI(V)>

set[T] -> set<ABI(T)>

tuple[T1, T2, ...] -> std::tuple<ABI(T1), ABI(T2), ...>

None (return value) -> void
```

Here, `ABI(T)` represents the ABI canonical form of the element type.

Important:

- The ABI of `list[str]` is `list<str>`, not `list<rc<str>>`.
- The ABI of `list[bytearray]` is `list<bytearray>`, not `rc<list<rc<bytearray>>>`.
- The ABI of `dict[str, int]` is `dict<str, int64>`.
- Even if the internal representation of `list[int]` is demoted to `rc<list<int64>>`, the ABI remains `list<int64>`.

### 5.3 Reference ABI Canonical Form

The following types may be treated as reference ABI rather than value ABI.

```text
Any / object -> object

user class C -> rc<C>

runtime objects requiring identity -> current runtime reference type
```

Examples:

- `Animal` -> `rc<Animal>`
- `Token` -> `rc<Token>`
- `object` / `Any` -> `object`

That is, `rc<>` is not completely banned; the meaning of this specification is that **it is not used in value type ABIs**.

### 5.4 C++ Argument Passing Form

The ABI type itself and how it is received in a C++ function declaration are considered separately.

Examples:

- ABI type: `list<bytearray>`
- C++ declaration form (recommended examples):
  - `const list<bytearray>& image`
  - `list<bytearray> image`
  - `list<bytearray>& image`

Which declaration form to use depends on mutation / copy cost / backend policy.
However, **the canonical form of the element type (`list<bytearray>`) must not be changed**.

## 6. ABI Adaptation Specification

### 6.1 Insertion Point for Adaptation

When lowering to the ABI-fixed boundary for an `@extern` function call or a `@abi`-specified helper call (e.g., in `CppLower`), do the following for each argument:

1. Obtain the internal type `Tin` of the actual argument (the type resulting from optimization)
2. Obtain the corresponding ABI type `Tabi` (determined by the fixed convention)
3. If `Tin` and `Tabi` match, pass as-is
4. If they do not match, insert `adapt(Tin -> Tabi)` and pass the result

### 6.2 Form of the Adaptation Function

ABI adaptation is expressed in generated code, for example, as follows:

```cpp
adapt_to_abi<list<bytearray>>(x)
```

The function name may be backend-implementation-dependent, but its role is limited to:

- Normalization from internal type to fixed ABI type
- No extraneous conversion inserted when it is a no-op

### 6.3 Required Adaptation Cases (Example: `list[bytearray]`)

For `Tabi = list<bytearray>`, at minimum the following cases must be handled:

#### Case 1: `Tin = list<bytearray>`

-> no-op

#### Case 2: `Tin = list<rc<bytearray>>`

-> Normalize each element to `bytearray` and construct `list<bytearray>`

#### Case 3: `Tin = rc<list<bytearray>>`

-> Strip the outer `rc<>` and construct `list<bytearray>`

#### Case 4: `Tin = rc<list<rc<bytearray>>>`

-> Strip the outer `rc<>` and also normalize each element to `bytearray`, constructing `list<bytearray>`

Any type not listed above may result in a compile error.

### 6.4 Additional Rules for `@abi(args={"x": "value"})`

For argument `x` in `@abi(args={"x": "value"})`, the backend must observe the following:

- The ABI canonical form is a value ABI
- However, the function declaration form in the target language may be a read-only borrow
- In C++, `const T&` may be used
- When it is possible to read-only borrow from an internal handle like `rc<list<T>>` to `const list<T>&`, unnecessary copies must not be inserted
- For internal types from which borrowing is not possible, insert an adapter fail-closed

### 6.5 Additional Rules for `@abi(args={"x": "value_mut"})`

For argument `x` in `@abi(args={"x": "value_mut"})`, the backend must observe the following:

- The ABI canonical form is a value ABI
- The callee may destructively modify this argument
- Adapter omission via read-only borrow must not be performed
- For targets where writable borrow cannot be expressed, insert an adapter / copy fail-closed

### 6.6 Additional Rules for `@abi(ret="value")`

The return value `ret="value"` fixes the returned value to the ABI canonical form value ABI.

This is a contract to prevent generated helpers from leaking internal ref-first representations to the outside.

### 6.6 Policy on Adaptation Cost

ABI adaptation may involve copying or reconstruction, but is accepted under the following principles:

- Prioritize fixing the ABI at external boundaries
- Insert adaptation "only when necessary" (no-op when types match)
- For I/O-type functions like `write_png`, boundary copies are unlikely to dominate
- If performance becomes a problem, design dedicated ABIs (e.g., span / pointer + size / writable buffer) individually

## 7. Generated C++ Declarations and Linking

### 7.1 How Declarations Are Provided

External function declarations are provided by one of the following:

- Method 1: The transpiler auto-generates forward declarations only for external functions that are used
- Method 2: Always include a common header where declarations are placed

Both are permitted, but Method 1 is recommended initially.

### 7.2 On `extern "C"`

The "ABI" in this specification means **Pytra's fixed boundary type**, not plain C ABI.

Therefore:

- `extern "C"` may be used to suppress name mangling
- However, when C++ types such as `list<bytearray>` or `dict<str, int64>` are included, this does not imply direct callability from C

`extern "C"` is optional but may be adopted for symbol name stability.

## 8. Specification Constraints

- An `@extern` function may have either a stub (`...` / `pass`) body or a Python-executable body on the Python side.
- However, in lowering to the target language, the function body is replaced by the external implementation.
- It is recommended that `@extern` functions have type annotations (arguments and return values).
- `@extern` may be restricted to top-level functions only.
- The constraint on where host-only imports (`import ... as __name`) may be referenced is required.

## 9. Example (`write_png`)

Python:

```python
from pytra import extern

@extern
def write_png(image: list[bytearray]) -> None: ...
```

ABI type (by convention):

- `image: list<bytearray>`
- Return value: `void`

Generated (conceptual):

```cpp
// example forward decl
extern "C" void pytra_utils_png_write_png(const list<bytearray>& image);

void callsite(...) {
    // image_in's type may fluctuate due to optimization (e.g., rc<list<rc<bytearray>>>)
    auto image_abi = adapt_to_abi<list<bytearray>>(image_in);
    pytra_utils_png_write_png(image_abi);
}
```

External implementation (C++ side):

```cpp
extern "C" void pytra_utils_png_write_png(const list<bytearray>& image) {
    // PNG write processing
}
```

## 10. Implementation Checklist

- Functions decorated with `@extern` can be detected from AST / EAST
- ABI type can be uniquely determined from Python type annotations
- External name `pytra_<module>_<func>` can be uniquely determined
- At call sites, `(Tin, Tabi)` can be compared and `adapt` inserted when necessary
- For representative examples like `list[bytearray]`, multiple internal types after demotion can be normalized to ABI
- External function declarations (forward decl or runtime header) can be generated

## 11. Runtime Directory Structure (Assumptions in This Specification)

The canonical runtime layout follows `docs/ja/spec/spec-runtime.md`.

Excerpting only the key points assumed in this specification, the current C++ runtime is organized in the following ownership lanes:

- `runtime/cpp/generated/{built_in,std,utils,compiler}/`
- `runtime/cpp/native/{built_in,std,utils,compiler}/`
- `runtime/cpp/generated/core/`
- `runtime/cpp/native/core/`

Ownership rules:

- Declarations / thin wrappers generated from the source of truth go in `generated/`
- Minimal native implementations that connect to OS / SDK / C++ standard library go in `native/`
- Low-level core splits ownership between `generated/core` and `native/core`

Notes:

- Ownership of the C++ module runtime is determined by directory, not by suffix.
- Suffix-based module runtimes in `src/runtime/cpp/{built_in,std,utils}` are legacy-closed and will not be reintroduced.
- For `core` as well, ownership is separated between `runtime/cpp/generated/core/` and `runtime/cpp/native/core/`; no checked-in `runtime/cpp/core/*.h` is held.
- The ABI concept itself is unchanged.
- For details, follow `docs/ja/spec/spec-runtime.md`, `docs/ja/plans/archive/20260307-p0-cpp-runtime-layout-generated-native.md`, and `docs/ja/plans/p0-cpp-core-ownership-split.md`.

## 12. `pytra.std.math` Example

`src/pytra/std/math.py` contains `@extern`. The structure for each language:

**C++** (declaration + linker binding):

- Hand-written implementation: `src/runtime/cpp/std/math.h` (the actual body of `@extern` functions)
- The emitter outputs only the declaration of `@extern` functions, combining the hand-written implementation via `#include`.

**JS / Dart / Julia, etc.** (delegation code generation):

- Hand-written native: `src/runtime/<lang>/std/math_native.<ext>` (host API connection)
- Emitter generates: `std/math.<ext>` (`@extern` functions delegate to `math_native`)
- Example (JS): `function sqrt(x) { return math_native.sqrt(x); }`

Common principles:

- ABI type is fixed at the value-type canonical form.
- The Python body of an `@extern` function is not adopted as the target implementation.
- `rc<>` is confined to the internal representation, not the ABI.
- The `generated/` / `native/` directory separation has been abolished (see spec-runtime.md §0.6a).
