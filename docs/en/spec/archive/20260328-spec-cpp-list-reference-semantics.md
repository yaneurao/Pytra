# C++ list Reference Semantics Specification (ref-first Canonical)

This document defines the canonical ref-first contract for mutable `list` in the C++ backend, along with the exception conditions under which deviations are permitted.

## 1. Purpose

- Fix the backend's internal representation of mutable `list` canonically as `rc<list<T>>`.
- Restrict generation of `list<T>` values to cases that are either "ABI adapter only" or "optimizer-proven".
- Clarify the alias / mutation / boxing boundary contracts that must be upheld in regression checks.

## 2. Scope

- In scope: `list[...]` as generated by the C++ runtime / C++ emitter.
- Out of scope: reference model migrations for `dict` / `set` / `str` (separate tasks).

## 3. Terminology

- `ref-first`: A policy of holding mutable values first in a shared-reference representation, and degenerating to a value type only on paths where safety has been proven.
- `ABI adapter`: An adapter that creates `list<T>` values temporarily, limited to `@extern` / `Any` / `object` / compatibility API boundaries.
- `optimizer-only value lowering`: Value-type demotion that the optimizer permits only when non-escape / alias / mutation / call graph / SCC conditions are all satisfied.
- `legacy value model`: Removed. Previously available as a rollback compatibility mode via `--cpp-list-model value`.
- `alias`: Sharing the same list, as in `b = a`.

## 4. Canonical Contract

- The default `list` model for `py2cpp` is `pyobj`.
- The canonical backend internal representation for mutable `list` is `rc<list<T>>`.
- Shared references must be maintained across alias, destructive updates, argument passing, return values, attribute storage, iteration, and subscript access.
- `list<T>` must not be directly generated via normal internal backend paths.
- The emitter must not choose a `list<T>` value for the following reasons:
  - Because the list is concretely typed
  - Because it is a local variable
  - Because no alias is visible
  - Because we want to shorten the appearance of a sample

## 5. Exception Conditions Permitting `list<T>` Values

### 5.1 ABI Adapter Only

- Argument/return value adapters for `@extern`.
- `Any/object` boxing / unboxing boundaries.
- Limited helpers for maintaining rollback compatibility APIs.

`list<T>` value helpers may remain for these purposes, but must not leak into the main backend paths.

### 5.2 Optimizer-Only Value Lowering

- `list<T>` value conversion is permitted only when the optimizer has proven it safe.
- Minimum conditions:
  - Mutation analysis
  - Alias analysis
  - Escape analysis
  - Call graph / SCC fixation
- Correctness must hold with ref-first intact; value conversion must be purely an optimization.

## 6. Alias / Mutation Contract

- `list` is shared by reference by default; `append/pop` after `b = a` must be mutually observable.
- The same list sharing must be maintained across function arguments, return values, and attribute storage.
- Boxing/unboxing at `Any/object` boundaries should prefer no-op compatibility (retaining the same underlying object).

## 7. Destructive Update Contract

- `append/pop/extend/clear` operate in-place on the list object.
- When shared aliases exist, the result of the update must be observable from all references.
- Only in the case of non-alias (distinct objects) are updates independent.

## 8. PyListObj Lifetime / Iterator Contract

- `PyListObj::py_iter_or_raise()` returns an iterator that references the owner list object, not a snapshot of the list values.
- The iterator holds a reference to the owner list's lifetime (and stops if the owner reference expires).
- Elements `py_append`-ed during iteration are reflected in the iteration results if they exist in the unvisited range.
- `py_try_len` / `py_truthy` are evaluated against the current state of the owner list object.

## 9. Fail-Closed Rules

- Do not apply optimization to paths that cannot be converted, have unknown types, or involve mixed external calls.
- Do not stack-allocate a list whose non-escape status cannot be proven.
- When semantics are unclear, fall to the heap/pyobj side.
- Do not convert a list to a value type when aliasing cannot be ruled out.
- Do not perform interprocedural value-type conversion before call graph / SCC summaries are finalized.

## 10. Handling Boundary Helpers

- `make_object(const rc<list<T>>& values)`, `obj_to_rc_list<T>`, `obj_to_rc_list_or_raise<T>`, `py_to_rc_list_from_object<T>`, `py_to_typed_list_from_object<T>` are to be used as boundary helpers only.
- These must not be repurposed for default representation selection within the emitter.
- Any design that broadly inserts `rc_list_ref(...)` or `list<T>(...)` for internal callsites, return values, or local variables is treated as an incomplete implementation under this specification.

## 11. Legacy Rollback Contract (Removed)

- `--cpp-list-model value` has been removed. pyobj (ref-first) is the only mode.
- The old `legacy value model` (value-typed list, copy assignment) has been fully deleted.

## 12. Acceptance Criteria

- Alias-expecting fixtures (`append/pop` after `a=b`) match Python behavior.
- `check_py2cpp_transpile` / C++ smoke / sample parity are satisfied.
- During the period when differences remain, the decision log in the planning document must record "case name · difference content".

## 13. Future Expansion

- This ref-first principle is not specific to `list`; in the future, the same approach will apply to `dict` / `set` / `bytearray` as well.
- However, the specific contracts and regression criteria in this document are limited to `list`.
