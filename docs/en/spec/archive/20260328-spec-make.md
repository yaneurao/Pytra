# Makefile Generation and One-Shot Build Specification

This document defines the operational specification for consolidating the C++ build pipeline into the `pytra` common CLI rather than calling `py2cpp.py` directly.

## 2026-02-24 Cross-Check Notes

- `src/pytra-cli.py` / `tools/gen/gen_makefile_from_manifest.py` / `./pytra` are implemented, providing the `--target cpp --build` pipeline via `./pytra`.
- The `manifest.json` contract for `--multi-file` and the `tools/build_multi_cpp.py` pipeline continue as specifications (`docs/ja/spec/spec-dev.md` / `docs/ja/spec/spec-tools.md`).
- This document is operated as an implemented specification; parts to be treated as future proposals are separated as additional tasks.

## 1. Decisions

- The user entry point is `./pytra` (extension-less launcher).
- The actual processing body is `src/pytra-cli.py` (run as `python src/pytra-cli.py`).
- `py2cpp.py` is maintained as a transpile backend and is not given build orchestration responsibilities.
- C++ builds are performed via canonical `manifest.json` + Makefile generation + `make` execution.
- `manifest.json` is not written directly by `ModuleEmitter`; it is treated as a build manifest output by `CppProgramWriter` which receives `ProgramArtifact`.
- Manual `PYTHONPATH` configuration is not needed (`./pytra` sets it internally).

## 2. Purpose

- Shorten manual commands and reduce transpilation through build to a single command.
- Leverage existing multi-file output and `manifest.json` to provide reproducible build steps.
- Establish an entry point that can be extended to a future multi-language integrated CLI (`--target` switching).

## 3. Non-Goals

- Immediately implementing build execution for all languages (start with `--target cpp` only).
- Generating IDE-specific project files (Visual Studio/Xcode).
- Auto-inferred builds that bypass `manifest.json`.

## 4. Entry Point Specification

### 4.1 `./pytra` Launcher

Place `pytra` (extension-less executable) at the repository root, performing the following:

1. Add `ROOT/src` to `PYTHONPATH`.
2. Execute `python3 src/pytra-cli.py "$@"`.

Rationale:

- Eliminates the need to type `PYTHONPATH=src ...` every time.
- Provides an executable form that does not conflict in name with `pytra.py` in input projects.

### 4.2 The Actual CLI

- The actual body is `src/pytra-cli.py`.
- Direct invocation form: `python3 src/pytra-cli.py ...`.

## 5. Common CLI Specification (v1)

### 5.1 Basic Form

```bash
./pytra INPUT.py --target cpp [OPTIONS]
```

### 5.2 Required/Supported Arguments in v1

- `INPUT.py`
- `--target cpp`
- `--output-dir DIR` (default: `out`)
- `--build` (build is executed only when specified)

### 5.3 C++ Build Options for `--build`

- `--compiler CXX` (default: `g++`)
- `--std STD` (default: `c++20`)
- `--opt FLAG` (default: `-O2`)
- `--exe NAME` (default: `app.out`)
- `--run` (optional: execute after successful build)

Notes:

- `--opt` in this specification refers to "C++ compilation flags".
- The optimization level for the generated code (the `py2cpp` `-O0..-O3`) may be separated as a distinct argument `--codegen-opt {0,1,2,3}` (uses default when unspecified).
- `--target cpp --codegen-opt 3` on `pytra-cli` means "max Pytra codegen route" — it selects backend emit via the linked-program optimizer, and is not simply an aggregate `-O3` passthrough.
- `--target cpp --codegen-opt 0/1/2` maintains the conventional route and is treated independently of compiler flags such as `--opt -O3`.

### 5.4 Constraints

- `--build` is valid only with `--target cpp`.
- `--compiler/--std/--opt/--exe/--run` are valid only when `--build` is specified.
- If `--build` is specified with a target other than `--target cpp`, the command exits with an error.

## 6. C++ Build Flow

The processing order for `./pytra ... --target cpp --build` is as follows.

1. For `--codegen-opt 3`: materialize the raw `EAST3` group, and generate C++ multi-file output from the linked module group through the linked-program optimizer.
2. For `--codegen-opt 0/1/2`: build `ProgramArtifact` via the conventional compat route.
3. `CppProgramWriter` generates the output tree and `manifest.json`.
4. Generate `Makefile` using `tools/gen/gen_makefile_from_manifest.py`.
5. Execute `make -f <Makefile>` to produce the binary.
6. Execute `make -f <Makefile> run` only when `--run` is specified.

Notes:

- In the current CLI implementation, `ProgramArtifact` is an internal concept, but `manifest.json` is treated as its concrete build artifact.
- The `SingleFileProgramWriter` for non-C++ backends does not require a build manifest. This document covers the manifest contract output by the C++ `CppProgramWriter`.

## 7. `manifest.json` Input Specification

`manifest.json` satisfies at least the following.

- `modules` is an array.
- Each element is an object, and `source` is a non-empty string.
- `include_dir` is a string (when unspecified, the `include` directory at the same level as the manifest may be used as the default).

Role in the linked-program phase:

- `manifest.json` is the C++ build serialization of `ProgramArtifact`.
- The generation responsibility belongs to `CppProgramWriter`; `CppEmitter` / `ModuleEmitter` do not directly generate `manifest.json`.
- `manifest.json` is the canonical source for build layout / runtime layout, not for language semantics.

Example:

```json
{
  "entry": "path/to/main.py",
  "include_dir": "out/include",
  "src_dir": "out/src",
  "modules": [
    {
      "module": "path/to/main.py",
      "label": "main",
      "header": "out/include/main.h",
      "source": "out/src/main.cpp",
      "is_entry": true
    }
  ]
}
```

## 8. Makefile Generation Specification

Using `tools/gen/gen_makefile_from_manifest.py`, accepting:

- Positional argument
  - `manifest`
- Options
  - `-o`, `--output`
  - `--exe`
  - `--compiler`
  - `--std`
  - `--opt`

The generated `Makefile` must include at minimum:

- Variables: `CXX`, `CXXFLAGS`, `INCLUDES`, `SRCS`, `OBJS`, `TARGET`
- Targets: `all`, `$(TARGET)`, `%.o: %.cpp`, `run`, `clean`

## 9. Error Contract

Exit with non-zero in the following cases:

- `manifest` does not exist.
- JSON parsing fails.
- `modules` is not an array.
- There is not a single valid `source` entry.
- `--compiler/--std/--opt/--exe/--run` is specified without `--build`.
- `--build` is specified with a target other than `--target cpp`.
- `make` is not found.

## 10. Acceptance Criteria

- `./pytra sample/py/01_mandelbrot.py --target cpp --output-dir out/mandelbrot` generates multi-file output.
- `./pytra sample/py/01_mandelbrot.py --target cpp --build --output-dir out/mandelbrot` executes transpilation, Makefile generation, and build sequentially.
- `./pytra sample/py/01_mandelbrot.py --target cpp --build --compiler g++ --std c++20 --opt -O3 --exe mandelbrot.out` reflects specified values in the `Makefile` and build.
- `./pytra sample/py/01_mandelbrot.py --target cpp --codegen-opt 3 --build --output-dir out/mandelbrot` selects the C++ max-opt route via the linked-program optimizer.
- `python3 tools/check/runtime_parity_check.py --targets cpp --case-root sample --cpp-codegen-opt 3 --east3-opt-level 2` stays green.
- `./pytra sample/py/01_mandelbrot.py --target rs --build` exits with an error as specified.
- A second `make -f out/mandelbrot/Makefile` execution results in a no-difference build (minimizing relinking/recompilation).

## 11. Phased Introduction

1. Phase 1: Add `tools/gen/gen_makefile_from_manifest.py`.
2. Phase 2: Add `src/pytra-cli.py` and implement `--target cpp --build`.
3. Phase 3: Add the root launcher `./pytra`, incorporating `PYTHONPATH` configuration.
4. Phase 4: Add `--run`, `--codegen-opt`, and `--jobs` if needed.

## 12. Practical Examples

### 12.1 Transpilation Only

```bash
./pytra sample/py/01_mandelbrot.py --target cpp --output-dir out/mandelbrot
```

### 12.2 One-Shot Build

```bash
./pytra sample/py/01_mandelbrot.py \
  --target cpp \
  --build \
  --output-dir out/mandelbrot
```

### 12.3 One-Shot Build (with explicit compiler settings)

```bash
./pytra sample/py/01_mandelbrot.py \
  --target cpp \
  --build \
  --output-dir out/mandelbrot \
  --compiler g++ \
  --std c++20 \
  --opt -O3 \
  --exe mandelbrot.out
```

### 12.4 Build + Run

```bash
./pytra sample/py/01_mandelbrot.py \
  --target cpp \
  --build \
  --run \
  --output-dir out/mandelbrot
```

## 13. Notes

- `out/` is operated as a local artifact directory and is not managed by Git.
- `py2cpp.py` is maintained as a backend and called from the common CLI.
- In the future, running `pip install -e .` + console script setup would allow `pytra ...` invocation without `./pytra`.

## 14. `pytra-cli` Responsibility Boundary (P0 Fixed Specification)

`src/pytra-cli.py` is limited to "common control at the entry point" and does not contain per-target build/run implementations.

- CLI body (`src/pytra-cli.py`)
  - Role: Argument normalization, input validation, profile resolution, common runner invocation.
  - Permitted: Passing target names to `toolchain`-side profiles.
  - Prohibited: Hard-coding target-specific compilers, runtimes, or execution commands.
- Backend profiles (`src/toolchain/misc/*`)
  - Role: Declare target-specific transpile/build/run contracts (required tools, output naming, auxiliary runtime files).
  - Permitted: Target-specific command / filename / extension definitions.
  - Prohibited: Re-implementing entry-point responsibilities such as CLI argument parsing or stdout/stderr control.
- Execution runner (common to CLI)
  - Role: Subprocess execution, stdout/stderr propagation, exit code handling, timeout management.
  - Permitted: Mechanically executing the command/cwd/env returned by the profile.
  - Prohibited: Branching on target name to rewrite to a different command.

### 14.1 Prohibitions (Monitored by CI)

- Must not add new `if/elif target == "...":` style branches to `src/pytra-cli.py`.
- Must not hard-code `<lang>`-specific runtime file paths (e.g., `py_runtime.kt`, `png.java`) in `src/pytra-cli.py`.
- Must not redundantly define target-specific build/run commands in peripheral tools such as `tools/check/runtime_parity_check.py` (unify through `pytra-cli`).
