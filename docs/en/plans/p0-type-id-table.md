# P0-TYPE-ID-TABLE: Linker-generated type_id table virtual module

Last updated: 2026-03-28
Status: Completed

## Background

The current C++ runtime has a hardcoded fixed-size array `g_type_table[4096]`, and the Go runtime has handwritten constants like `PYTRA_TID_VALUE_ERROR = 12`. Although the linker determines type_ids, there is no mechanism to pass those values to the runtime, so the system works only because the numbers happen to match by coincidence.

The `pytra.built_in.type_id_table` virtual module designed in spec-type_id.md §7 will be generated by the linker, and emitters will map it as ordinary EAST3, thereby eliminating hardcoded type_id tables across all languages.

## Subtasks

1. [ID: P0-TID-TABLE-S1] Define `pytra_isinstance(actual_type_id: int, tid: int) -> bool` in pure Python under `pytra/built_in/` — a predicate function using `id_table[tid * 2] <= actual_type_id <= id_table[tid * 2 + 1]`
2. [ID: P0-TID-TABLE-S2] Implement `pytra.built_in.type_id_table` virtual module generation in the linker — generate `id_table: list[int]` and per-type TID constants (e.g. `VALUE_ERROR_TID: int = 3`) as EAST3 and include them in the link output
3. [ID: P0-TID-TABLE-S3] When the linker lowers isinstance to `pytra_isinstance(x.type_id, TID_CONSTANT)`, insert an import of `pytra.built_in.type_id_table` into the target module's `meta.import_bindings`
4. [ID: P0-TID-TABLE-S4] Retire the hardcoded `g_type_table[4096]` from the C++ runtime and migrate to using the linker-generated `pytra.built_in.type_id_table`
5. [ID: P0-TID-TABLE-S5] Retire the handwritten TID constants (e.g. `PYTRA_TID_VALUE_ERROR = 12`) from the Go runtime and migrate to using the linker-generated module
6. [ID: P0-TID-TABLE-S6] C++/Go parity check for fixture + sample

## Acceptance Criteria

1. `pytra.built_in.type_id_table` is included in the linker's link output
2. isinstance is output in generated code as `pytra_isinstance(x.type_id, VALUE_ERROR_TID)`
3. No hardcoded fixed-size array (`g_type_table[4096]`) exists in the C++ runtime
4. No handwritten TID constants exist in the Go runtime
5. Existing fixture + sample C++/Go parity is maintained

## Decision Log

- 2026-03-28: Code review of C++ runtime revealed the hardcoded `g_type_table[4096]`. Go also has handwritten constants. Identified the problem: the linker determines type_ids but has no mechanism to pass them to the runtime.
- 2026-03-28: Designed an approach where the linker generates `pytra.built_in.type_id_table` as a virtual module, placing `id_table: list[int]` and TID constants in EAST3. Emitters map it as ordinary EAST3.
- 2026-03-28: The isinstance predicate takes the form `pytra_isinstance(actual_type_id, tid)` with 1 type argument. `id_table[tid * 2]` gives min, `id_table[tid * 2 + 1]` gives max.
- 2026-03-28: The sole source of truth for helper generation of `id_table` / TID constants is the output of `build_type_id_table()`; the proposal to hardcode min/max inside the linker is withdrawn.
- 2026-03-28: The Go backend emits only `pytra.built_in.type_id_table` and references the `pytra_isinstance` body as a helper in `py_runtime.go`, maintaining parity.
